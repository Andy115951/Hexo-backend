## 一句话成就

把零散脚本测试升级为分层自动化测试体系，并建立“覆盖治理 + 稳定性治理 + 调试闭环”的持续质量保障机制。

---

## 成就卡片

- 成就编号：A04
- 时间范围：2025-2026
- 业务域：自动化测试平台 / 发布质量保障
- 我的角色：框架设计 + 用例建设 + 稳定性治理 owner
- 影响对象：测试效率、回归可信度、发布前质量门禁

---

## 背景与问题定义

### 1) 业务场景

功能迭代加快后，手工回归成本持续上升。测试任务跨模块、跨端、跨环境，导致“漏测、误报、复现慢”的问题不断放大。

### 2) 原始痛点

1. 测试脚本分散，选择器与业务逻辑耦合，维护成本高。
2. 用例覆盖缺少台账，优先级管理弱，补测常凭感觉。
3. flaky 问题频发，失败原因难快速定位。
4. 本地与 CI 行为不一致，结果可信度不足。

### 3) 影响评估

- 交付风险：发版前验证不完整。
- 效率风险：排障时间高于用例编写时间。
- 资产风险：测试经验难沉淀，人员变动影响大。

---

## 目标与验收标准

### 1) 目标

1. 建立可扩展、可维护的分层测试框架。
2. 把核心业务路径纳入可持续回归链路。
3. 降低 flaky 干扰，提升失败定位效率。
4. 打通本地到 CI 的统一执行标准。

### 2) 验收标准

- 框架具备清晰分层：`core / selectors / actions / tests`。
- 核心场景具备冒烟+回归覆盖，并可按优先级扩展。
- 用例失败能通过统一调试策略快速定位。
- CI 产出可追溯报告并支持复盘。

---

## 约束与边界

- 约束 1：UI 迭代快，选择器稳定性天然受挑战。
- 约束 2：桌面应用自动化受启动模式和环境差异影响大。
- 约束 3：测试资源有限，必须按优先级治理覆盖范围。
- 非目标：本阶段不追求“100% 自动化覆盖”，优先保障核心链路稳定。

---

## 核心原理（为什么这样设计）

1. 测试代码和业务 UI 要解耦  
把选择器和动作封装分层，才能在 UI 变更时把影响面控制在最小范围。

2. 稳定性优先于覆盖率  
不稳定的高覆盖会制造噪音，先把核心链路跑稳，再做广度扩展。

3. 等待策略要以状态为中心  
固定 sleep 只能“碰运气”，状态驱动等待才可迁移、可复现。

4. 测试治理本质是风险治理  
覆盖运营应围绕风险优先级，而不是围绕用例总数。

### 原理流转图（简化）

```text
需求变更进入测试池
  -> 映射到分层框架（selector/action/test）
    -> 进入覆盖状态台账（稳定/风险/缺失）
      -> 执行回归（基线模式）
        -> 失败分流（环境问题 vs 业务问题）
          -> 修复后回归并更新台账
```

---

## 我的关键动作

## 1) 框架分层重构

- 把原本耦合脚本拆成核心层、选择器层、动作层、测试层。
- 将页面细节从业务步骤中剥离，降低 UI 变化引发的连锁修改。
- 建立统一入口类，减少用例重复代码。

## 2) 覆盖度台账化治理

- 建立覆盖矩阵：已覆盖、已覆盖但风险高、框架有能力但无测试、未覆盖。
- 引入优先级策略：先保核心回归，再逐步覆盖边缘能力。
- 让补测计划有依据，不再依赖临场判断。

## 3) 稳定性治理与调试能力建设

- 把固定超时等待改为状态驱动等待，减少误失败。
- 增加双模式调试策略，区分“环境问题”和“业务问题”。
- 沉淀失败定位路径：现象 -> 环境 -> 选择器 -> 业务逻辑 -> 断言。

## 4) CI 闭环建设

- 统一配置入口，支持环境变量控制关键参数。
- 打通执行、报告、归档路径，确保结果可追溯。
- 配套文档化，保证新成员可快速接入并复用。

---

## 难点、风险与应对

### 1) 难点

- UI 元素动态变化导致脆弱选择器反复失效。
- 异步行为波动导致“偶发红灯”难解释。
- 不同运行环境造成“本地过、CI 挂”的不一致。

### 2) 风险与应对

- 风险：为追求覆盖率引入大量低价值用例。  
  应对：优先级治理，先关键路径后边缘能力。
- 风险：调试链路不统一导致定位效率低。  
  应对：统一故障分层诊断流程与排查文档。

### 3) 关键突破

- 从“脚本堆砌”升级为“框架化治理”。
- 从“跑通即完成”升级为“稳定可持续运行”。

---

## 结果与影响

### 1) 短期结果

1. 核心路径自动化回归可持续执行。
2. 用例维护成本下降，新增场景接入速度提升。

### 2) 中期价值

1. 失败定位从随机排查变成结构化排查。
2. 发布前验证更可控，质量门禁更加稳健。

### 3) 长期复用价值

1. 测试资产从个人经验沉淀为团队资产。
2. 框架和治理方法可迁移到其他端与模块。

## 关键代码与命令片段（脱敏示例）

```ts
// actions/message.actions.ts
export class MessageActions {
  constructor(private page: Page, private selectors: Selectors) {}

  async sendMessage(text: string) {
    await this.page.locator(this.selectors.input).fill(text);
    await this.page.locator(this.selectors.sendBtn).click();
    await this.waitUntilResponseStable();
  }

  async waitUntilResponseStable() {
    await expect(this.page.locator(this.selectors.streamingFlag)).toBeHidden({ timeout: 20000 });
    await expect(this.page.locator(this.selectors.lastMessage)).toBeVisible();
  }
}
```

## 可复用经验

1. 自动化测试先保证“关键路径稳定”，再追求广覆盖。
2. 分层不是形式，而是维护成本控制的核心抓手。
3. flaky 治理必须组合策略：等待机制、调试模式、排障文档。
4. 覆盖治理要有台账和优先级，否则无法持续演进。

---

## 述职/面试可直接复用的话术

“我做的不只是增加用例数量，而是把自动化测试做成了可持续体系：架构上分层、治理上有覆盖台账、运行上有稳定性策略、交付上打通了 CI 闭环。这个体系能长期支撑迭代质量，而不是短期冲刺。”
