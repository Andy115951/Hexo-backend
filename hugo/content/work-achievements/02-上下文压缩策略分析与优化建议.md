## 一句话成就

把“配置较小上下文仍频繁触发压缩”的争议问题，从现象级讨论转化为机制级分析，并给出可分阶段落地的优化方案。

---

## 成就卡片

- 成就编号：A02
- 时间范围：2026-01
- 业务域：会话架构 / 体验稳定性
- 我的角色：机制分析 + 方案设计 + 优化路径 owner
- 影响对象：产品体验、存储成本、研发实现路径

---

## 背景与问题定义

### 1) 业务场景

在长会话中，用户设置了较小上下文窗口，但系统仍持续触发压缩，导致用户认知与系统行为不一致，反馈集中在“像 bug、不可理解、不可预期”。

### 2) 原始痛点

1. 团队讨论停留在“配置是否生效”，缺少统一机制视角。
2. 业务、产品、研发对“压缩触发条件”理解不一致。
3. 只有现象没有模型，难以制定可执行优化方案。

### 3) 影响评估

- 体验层：用户难理解为什么被压缩，信任感下降。
- 工程层：修改点不明确，容易“修一处坏一处”。
- 运营层：问题反复出现，沟通成本高。

---

## 目标与验收标准

### 1) 目标

1. 明确压缩触发链路和配置作用边界。
2. 给出至少两种可比较策略，并明确收益与代价。
3. 输出分阶段落地方案，支持逐步验证与回滚。

### 2) 验收标准

- 团队对“请求窗口控制”与“全局存储压缩”定义一致。
- 每种策略都给出适用场景、风险和验证指标。
- 优化方案包含实现优先级、观测点和回退条件。

---

## 约束与边界

- 约束 1：压缩策略必须兼顾体验、成本、性能三方目标。
- 约束 2：会话历史具有长期价值，不能简单粗暴删除。
- 约束 3：系统需支持可恢复与可追溯，不可只追求体积最小。
- 非目标：本次不直接重写全量实现，先完成机制拆解与方案收敛。

---

## 核心原理（为什么这样设计）

1. 请求窗口预算与历史存储预算必须分离  
“本次发送多少”与“长期保留多少”是两类约束，混在一起必然导致认知混乱。

2. 压缩要遵循冷热分层  
活跃区保真、半活跃区摘要化、冷区批量治理，才能同时兼顾体验和成本。

3. 压缩必须可恢复  
通过摘要锚点保留追溯路径，避免压缩后语义不可还原。

4. 触发机制需要滞回窗口  
避免在阈值附近频繁压缩，减少体验抖动。

### 原理流转图（简化）

```text
会话新增内容
  -> 判断发送窗口预算
    -> 超预算：窗口内压缩
    -> 未超预算：检查存储预算
      -> 超预算：渐进式压缩（冷区优先）
      -> 未超预算：维持原状态
  -> 写入可恢复锚点
```

---

## 我的关键动作

## 1) 从“抱怨描述”到“机制拆解”

我先把问题拆成两条链路：

1. 发送链路：控制本次请求携带哪些上下文。
2. 存储链路：控制整段历史如何压缩和保留。

关键结论：用户配置通常只影响发送窗口，不必然等于存储不压缩。这是认知偏差的核心来源。

## 2) 进行策略对照，避免单结论误导

我做了三种策略对照：

1. 全局压缩优先  
优势：控体积强。  
代价：可能压到用户仍关心内容。

2. 仅窗口内压缩  
优势：直觉一致性高。  
代价：历史膨胀风险高，长期恢复能力弱。

3. 渐进式压缩（推荐）  
优势：活跃区优先保护，旧历史分段治理。  
代价：实现复杂度更高，需要更多观测指标。

## 3) 定义渐进式压缩落地框架

- 第一层：活跃消息区轻量压缩，保证当前对话连续性。
- 第二层：半活跃区做摘要化与锚点保留。
- 第三层：冷历史区按阈值批量治理。

同时定义了“可恢复锚点”原则，避免压缩后彻底不可追溯。

## 4) 补齐可验证指标

我给出三类验收指标建议：

1. 体验一致性：压缩行为是否与用户配置预期一致。
2. 存储治理：历史增长曲线是否可控。
3. 可恢复性：压缩后是否可通过锚点恢复关键上下文。

---

## 难点、风险与应对

### 1) 难点

- 同一“压缩”在不同层语义不同，沟通极易错位。
- 只优化单一目标会导致其他目标退化。

### 2) 风险与应对

- 风险：过度压缩影响对话连续性。  
  应对：活跃区优先保护 + 渐进式策略。
- 风险：策略复杂后难以解释。  
  应对：分层术语标准化 + 触发链路图可视化。

### 3) 关键突破

- 把“配置争议”转为“机制分层问题”。
- 把“拍脑袋优化”转为“指标驱动优化”。

---

## 结果与影响

### 1) 短期结果

1. 团队在机制定义上形成统一语言和共识。
2. 优化方向从“猜原因”变成“按链路改进”。

### 2) 中期价值

1. 后续实现可按阶段推进，降低一次性改动风险。
2. 验收标准清晰，避免重复争论“到底算不算修好”。

### 3) 长期复用价值

1. 上下文治理形成可复用方法论，可迁移到其他长会话系统。
2. 为后续容量规划、体验优化提供统一框架。

## 关键代码与命令片段（脱敏示例）

```ts
type CompressDecision = {
  shouldCompress: boolean;
  reason: 'window_limit' | 'storage_pressure' | 'manual_trigger' | 'none';
};

export function decideCompression(input: {
  promptTokens: number;
  windowLimit: number;
  historyBytes: number;
  storageThreshold: number;
  forceCompress?: boolean;
}): CompressDecision {
  if (input.forceCompress) return { shouldCompress: true, reason: 'manual_trigger' };
  if (input.promptTokens > input.windowLimit) return { shouldCompress: true, reason: 'window_limit' };
  if (input.historyBytes > input.storageThreshold) return { shouldCompress: true, reason: 'storage_pressure' };
  return { shouldCompress: false, reason: 'none' };
}
```

```ts
// 渐进式压缩：活跃区保护 + 冷区批量治理
export function progressiveCompress(messages: Message[]): Message[] {
  const active = messages.slice(-30);      // 活跃区：尽量不动
  const warm = messages.slice(-120, -30);  // 半活跃区：摘要化
  const cold = messages.slice(0, -120);    // 冷区：批量压缩

  const warmSummary = summarizeChunks(warm, { keepAnchors: true });
  const coldSummary = summarizeChunks(cold, { keepAnchors: true, aggressive: true });

  return [...coldSummary, ...warmSummary, ...active];
}
```

## 可复用经验

1. 配置问题先区分“用户语义”和“系统语义”。
2. 策略优化要给对照方案，不做单路径结论。
3. 机制设计必须同步定义验收指标与回退条件。
4. 先统一术语，再推进实现，能显著降低沟通损耗。
