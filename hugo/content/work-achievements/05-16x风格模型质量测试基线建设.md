## 一句话成就

把“模型输出质量靠感觉”的评估方式，升级为“统一任务 + 自动评分 + 稳定性统计”的可复现评测基线。

---

## 成就卡片

- 成就编号：A05
- 时间范围：2025-2026
- 业务域：模型质量评测 / 选型与回归
- 我的角色：评测规则设计 + 评分引擎实现 + 数据解释 owner
- 影响对象：模型选型、版本回归、质量监控

---

## 背景与问题定义

### 1) 业务场景

模型迭代频繁，团队需要在多个候选模型间快速比较质量，并判断某版本是否发生能力退化。

### 2) 原始痛点

1. 评估标准不统一，结论高度依赖个人偏好。
2. 只看单次输出容易误判，无法识别稳定性问题。
3. 有结果无解释，难形成可执行改进建议。

### 3) 影响评估

- 决策风险：模型选型容易受主观印象影响。
- 协作风险：同一输出不同人结论差异大。
- 复盘风险：无法回答“这次好坏到底差在哪”。

---

## 目标与验收标准

### 1) 目标

1. 定义统一任务与评分规则，保证对比公平。
2. 自动化输出分数、问题类型、质量排名。
3. 引入多次运行统计，识别稳定性差异。

### 2) 验收标准

- 同一输入和目标输出可对多个模型批量运行评测。
- 结果包含分数、差异量、问题分级，而不仅是“通过/失败”。
- 支持多次尝试归并，输出平均分、最好/最差、标准差。
- 评测结果可通过脚本重复执行并复现。

---

## 约束与边界

- 约束 1：评测任务需足够聚焦，避免任务过大导致解释困难。
- 约束 2：评分必须兼顾准确性与执行成本，适配日常迭代节奏。
- 约束 3：评测不等于完整能力画像，本次聚焦文本清洗任务质量。
- 非目标：不在本阶段做全场景 benchmark，只先把“高频关键任务”做精。

---

## 核心原理（为什么这样设计）

1. 同任务、同标准、同输入才可比较  
模型评测不是“看输出印象”，而是受控实验，必须保证输入和规则一致。

2. 评分需要可解释拆解  
只有总分不够，必须能分解到错误类型，才能支持后续优化。

3. 单次结果不代表真实能力  
平均分 + 波动性（sigma）比峰值更接近生产可靠性。

4. 评测系统要支持持续复跑  
能在日常迭代中低成本重复执行，评测结论才有长期价值。

### 原理流转图（简化）

```text
统一输入与规则
  -> 多模型执行同任务
    -> 自动评分（分数 + 问题分类）
      -> 多次运行归并（avg/best/worst/sigma）
        -> 生成排名与改进建议
```

---

## 我的关键动作

## 1) 固化任务与评分规则

- 明确任务规则：去 frontmatter、去 import、去表格/组件、展开代码块与链接、保留正文语义等。
- 建立 rubrics：内容完整性为主，结构残留和格式问题为扣分项。
- 增加“短实现/高效率”正向加分与“剩余差异”惩罚，避免单点优化刷分。

## 2) 构建自动评分引擎

- 基于字符差异、行差异做基础量化。
- 实现问题类型识别（critical / major / minor）并映射扣分影响。
- 输出每个模型的缺陷列表，便于直接定位改进方向。

## 3) 增加稳定性统计维度

- 设计同模型多次尝试归并机制（按 base name 聚合）。
- 统计平均分、最高分、最低分、标准差（sigma）。
- 排名优先看平均分和稳定性，而不是单次峰值。

## 4) 形成可执行操作面

- 提供标准、详细、单模型、评分视图等多模式命令。
- 支持导入不同模型实现，统一进入同一评测流程。
- 让评测从一次性实验变成持续可跑的工程能力。

---

## 难点、风险与应对

### 1) 难点

- 单任务评测容易高分，但不代表稳定可靠。
- 纯分数无法表达“错在哪”，改进路径不清晰。

### 2) 风险与应对

- 风险：单次偶然高分误导选型。  
  应对：强制引入多次运行与稳定性统计。
- 风险：评分细则过粗导致区分度不足。  
  应对：分级问题识别 + 差异惩罚组合评分。

### 3) 关键突破

- 从“主观比输出”升级为“规则化评测”。
- 从“看最高分”升级为“看平均分 + 波动性”。

---

## 结果与影响

### 1) 短期结果

1. 形成统一的模型质量基线，日常可快速对比。
2. 输出可解释评分报告，便于快速定位缺陷类型。

### 2) 中期价值

1. 模型选型讨论从经验判断转为证据驱动。
2. 版本回归可用同一评测流程识别能力波动。

### 3) 长期复用价值

1. 评分引擎可复用到其他高频任务。
2. 稳定性视角可作为模型上线前质量门槛的一部分。

## 关键代码与命令片段（脱敏示例）

```ts
function calculateScore(issues: Issue[], diffChars: number, diffLines: number): number {
  let score = 100;
  for (const issue of issues) score += issue.impact; // impact 为负值时扣分

  const categorizedImpact = issues.reduce((sum, it) => sum + Math.abs(it.impact), 0);
  const remainingPenalty = Math.max(0, diffChars - categorizedImpact) * 0.1;
  score -= remainingPenalty;

  if (diffLines === 0 && score > 90) score += 1; // 格式稳定性奖励
  return Math.max(0, Math.min(100, Number(score.toFixed(1))));
}
```

```ts
// 多次尝试归并：看平均分和波动，而不是只看最高分
function summarizeAttempts(scores: number[]) {
  const avg = scores.reduce((a, b) => a + b, 0) / scores.length;
  const variance = scores.reduce((sum, s) => sum + (s - avg) ** 2, 0) / scores.length;
  return {
    avg: Number(avg.toFixed(1)),
    best: Math.max(...scores),
    worst: Math.min(...scores),
    sigma: Number(Math.sqrt(variance).toFixed(2))
  };
}
```

## 可复用经验

1. 模型评测必须先统一任务边界，才能公平比较模型。
2. 评分体系要同时回答两个问题：好不好、差在哪。
3. 稳定性比峰值更重要，尤其在生产选型阶段。
4. 可复现脚本是评测体系长期有效的核心。
