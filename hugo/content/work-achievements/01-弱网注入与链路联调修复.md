把“偶发网络失败难复现”的问题，升级为“可重复、可分档、可验证”的弱网联调体系，并沉淀成团队可直接复用的脚本与操作手册。

- 我的角色：方案设计 + 脚本落地 + 排障 owner
- 影响对象：测试、研发、发布前验证流程

---

## 背景与问题定义

### 1) 业务场景

在稳定性验证阶段，需要对多条上游链路并行做弱网注入，用于复现“偶发超时、连接中断、响应抖动”等问题，并对比修复前后行为差异。

### 2) 原始痛点

1. 弱网注入可偶尔生效，但无法稳定复现，导致问题难以复盘。
2. 多链路并行场景下，单点配置常互相干扰，结果不可解释。
3. 出现失败时无法快速判断根因属于配置、网络、权限还是环境冲突。
4. 操作高度依赖个人经验，难以交接和团队复用。

### 3) 影响评估

- 复现效率低：同类问题反复人工尝试。
- 定位周期长：链路复杂导致排查顺序混乱。
- 验证可信度低：缺少统一基线与证据标准。

---

## 目标与验收标准

### 1) 目标

1. 建立多链路并行可控的弱网注入能力。
2. 形成标准档位，保证不同人执行结果一致。
3. 建立可追溯证据链，支持“复现-修复-回归”闭环。

### 2) 验收标准

- 能在 `normal / mild / flaky / harsh` 四档之间稳定切换。
- 能同时对多目标链路进行状态查询、档位切换和恢复。
- 每次联调都能留下三类证据：工具状态、连通性结果、日志轨迹。
- 出现失败时可按固定排查顺序在可接受时间内收敛根因。

---

## 约束与边界

- 约束 1：本机网络环境和代理策略会影响链路可达性。
- 约束 2：不同运行模式（容器/宿主）存在实现差异。
- 约束 3：低端口、进程残留、DNS 路径等会产生叠加噪音。
- 非目标：本次不追求全量网络协议仿真，只聚焦核心链路弱网复现能力。

---

## 核心原理（为什么这样做）

1. 控制面与数据面分离  
控制脚本只负责“配置与状态”，验证脚本只负责“连通性与日志”，避免职责混淆。

2. 弱网档位标准化  
把参数组合抽象成固定 profile，保证不同人、不同时间执行结果可比。

3. 证据三联单原则  
任何结论必须同时满足状态证据、连通性证据、日志证据，防止“假恢复”。

4. 排障顺序优先于排障技巧  
先确定问题所在层，再使用技巧优化，能显著降低误判率。

### 原理流转图（简化）

```text
选择弱网档位
  -> 注入到目标链路
    -> 采集三类证据（状态/连通性/日志）
      -> 证据一致：判定有效
      -> 证据不一致：进入分层排障
```

---

## 我的关键动作

## 1) 搭建可执行控制平面

- 把启动、下线、切档、状态查询统一到脚本入口。
- 把弱网参数抽象为标准档位，降低临场调参误差。
- 支持多目标并行操作，避免逐个链路手工切换。

## 2) 建立“链路分层排障法”

我把排障拆成固定顺序，避免盲查：

1. 端口与进程：确认无冲突、无残留。
2. 网络入口：确认监听地址与回环可用。
3. 代理配置：确认目标映射完整且无漏配。
4. DNS 路径：确认未被异常策略劫持导致假目标。
5. 运行模式：比对容器与宿主模式差异。
6. 权限边界：确认低端口与系统权限限制。

## 3) 固化“证据化验证”

每次验证必须同时满足：

1. 状态证据：代理是否处于预期档位。
2. 连通性证据：目标请求在预期扰动下是否出现可解释行为。
3. 日志证据：建立连接、注入扰动、收尾恢复完整可追踪。

## 4) 输出可复用 runbook

- 把问题-根因-动作-结论整理为标准排障记录。
- 给出“先做什么、再做什么、最后如何验收”的执行顺序。
- 形成新成员可直接跟做的操作说明。

---

## 难点、风险与应对

### 1) 难点

- 多因素耦合导致“看起来像网络问题，实际是环境冲突”。
- 不同模式的行为差异让一次性结论难复用。

### 2) 风险与应对

- 风险：档位切换成功但并未真正命中链路。  
  应对：强制加入定向连通性校验。
- 风险：恢复流程不完整造成后续环境污染。  
  应对：把恢复动作纳入脚本标准流程。

### 3) 关键突破

- 从“经验式试错”升级到“结构化排障顺序”。
- 从“单次修好”升级到“可重复复现 + 可审计证据”。

---

## 结果与影响

### 1) 短期结果

1. 弱网场景从偶发可复现提升为可稳定复现。
2. 联调过程有统一入口，操作一致性显著提升。

### 2) 中期价值

1. 问题定位不再依赖某个熟手，协作效率更高。
2. 修复验证可做前后对比，回归结论可信度更高。

### 3) 长期复用价值

1. 弱网测试能力成为可持续复用的工程资产。
2. 后续同类链路问题可复用同一套方法快速收敛。

## 关键代码与命令片段（脱敏示例）

```bash
#!/usr/bin/env bash
set -euo pipefail

PROFILE="${1:-normal}"   # normal|mild|flaky|harsh
TARGET="${2:-all}"       # all|provider-a|provider-b

apply_profile() {
  local profile="$1"
  local target="$2"
  ./scripts/network-chaos.sh up "$target"
  ./scripts/network-chaos.sh profile "$profile" "$target"
  ./scripts/network-chaos.sh status "$target"
}

verify_path() {
  local endpoint="$1"
  curl -sS --max-time 8 "$endpoint/healthz" | jq .
}

apply_profile "$PROFILE" "$TARGET"
verify_path "http://127.0.0.1:8474"
```

```bash
# 标准联调顺序（命令化 runbook）
./scripts/network-chaos.sh down all
./scripts/network-chaos.sh up all
./scripts/network-chaos.sh profile mild all
./scripts/network-chaos.sh status all

# 证据留存：状态 + 连通性 + 日志
curl -sS http://127.0.0.1:8474/proxies | jq .
tail -n 80 /tmp/network-chaos.log
```

## 可复用经验

1. 联调任务先画链路分层图，再进入排障。
2. 先验证路径正确性，再做性能或权限优化。
3. 每个弱网档位都要有 `normal` 对照基线。
4. 结论必须附证据，不做“感觉修好了”的判断。
