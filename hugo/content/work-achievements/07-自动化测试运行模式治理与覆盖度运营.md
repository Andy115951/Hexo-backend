## 一句话成就

把“能跑但不稳”的自动化测试升级为“运行模式清晰、覆盖状态透明、CI 闭环可追溯”的可持续测试运营体系。

---

## 成就卡片

- 成就编号：A07
- 时间范围：2026-03 ~ 2026-04
- 业务域：自动化测试治理 / 质量运营
- 我的角色：运行策略设计 + 覆盖治理 + CI 闭环落地
- 影响对象：测试执行效率、回归可信度、发布前质量决策

---

## 背景与问题定义

### 1) 业务场景

随着测试规模增长，自动化体系开始出现“可执行但不稳定”的典型问题。核心矛盾不再是“有没有测试”，而是“跑出来能不能信”。

### 2) 原始痛点

1. 启动链路受环境影响大，失败原因混杂。
2. 覆盖状态不透明，缺项与风险项混在一起。
3. 本地调试和 CI 执行路径脱节，结果难对齐。

### 3) 影响评估

- 执行风险：同一用例跨环境结果不一致。
- 治理风险：覆盖率数字高但关键路径仍可能漏测。
- 协作风险：排障时间占比过高，反馈周期被拉长。

---

## 目标与验收标准

### 1) 目标

1. 明确运行模式分工，降低环境噪音对结果的影响。
2. 建立覆盖状态台账，形成可执行补测优先级。
3. 打通本地到 CI 的统一报告链路，提升复盘效率。

### 2) 验收标准

- 运行策略明确：基线模式、调试模式、失败回退规则清晰。
- 覆盖台账能区分“已覆盖/有风险/缺失项/框架能力缺口”。
- CI 结果可归档并可回溯到具体用例与执行上下文。
- 团队可按优先级持续推进补测，而不是临时补丁式治理。

---

## 约束与边界

- 约束 1：桌面端自动化受启动机制和系统环境影响显著。
- 约束 2：覆盖治理必须在有限资源下做取舍，不可全量并行推进。
- 约束 3：测试可信度建设是长期工程，不可能一次改造完成。
- 非目标：本阶段不追求一次性消灭全部 flaky，用分层治理持续收敛。

---

## 核心原理（为什么这样设计）

1. 执行基线与诊断链路分离  
“稳定回归”与“问题定位”目标不同，必须用不同模式服务不同目标。

2. 覆盖率不等于风险覆盖  
治理要从“覆盖数量”升级为“覆盖状态 + 风险权重”。

3. 质量治理要闭环  
用例状态、执行结果、CI 报告、补测计划必须能串起来。

4. 稳定性是持续运营问题  
不是一次修复，而是台账化、节奏化、可追踪地持续收敛。

### 原理流转图（简化）

```text
回归任务进入队列
  -> 选择执行模式（launch 基线 / cdp 调试）
    -> 执行用例
      -> 结果分层（稳定 / 风险 / 缺失 / 债务）
        -> CI 报告归档
          -> 生成补测优先级（P0/P1/P2）
            -> 下一轮回归
```

---

## 我的关键动作

## 1) 运行模式治理：把“执行”与“定位”分开

我定义了双模式策略：

1. `launch` 模式：作为稳定回归基线，保证发布前可重复执行。
2. `cdp` 模式：作为调试与可观测增强手段，用于快速定位问题。

并补充了模式切换与失败回退原则，避免“连不上就整体停摆”。

## 2) 覆盖度运营：从总数视角转为状态视角

我把覆盖治理拆为四类状态并运营：

1. 已覆盖且稳定
2. 已覆盖但存在风险（需复测）
3. 框架已有能力但缺测试用例
4. 尚未覆盖

这让补测计划从“拍脑袋”变成“按状态和优先级推进”。

## 3) 优先级策略落地

- P0：唯一未覆盖关键项与高风险关键链路优先补齐。
- P1：框架已有能力但未落用例的场景尽快补齐。
- P2：体验增强类和低风险边缘场景按节奏推进。

通过优先级明确，测试投入和业务风险之间形成可解释关系。

## 4) CI 闭环与文档化

- 统一配置项，支持环境变量控制执行路径。
- 输出报告归档策略，支持问题追溯与对比复盘。
- 把本地验证、CI 配置、故障定位方法写成可执行文档。

---

## 难点、风险与应对

### 1) 难点

- 环境问题和业务问题常被混淆，导致误判。
- 覆盖率高不代表风险低，治理需要更细颗粒度视角。

### 2) 风险与应对

- 风险：调试模式过度使用影响基线可信度。  
  应对：明确 `launch` 为基线，`cdp` 仅用于定位。
- 风险：覆盖台账更新滞后导致决策失真。  
  应对：将状态更新纳入回归例行流程。

### 3) 关键突破

- 从“会跑测试”升级为“治理测试”。
- 从“覆盖率单指标”升级为“覆盖状态 + 风险优先级”。

---

## 结果与影响

### 1) 短期结果

1. 运行策略清晰后，执行稳定性和定位效率明显提升。
2. 覆盖缺口可见，补测计划具备可执行顺序。

### 2) 中期价值

1. 本地调试和 CI 结果的一致性更高，回归可信度提升。
2. 团队围绕台账协作，沟通成本显著下降。

### 3) 长期复用价值

1. 覆盖运营方式可迁移到其他测试域。
2. 质量治理从“临时响应”转向“持续运营”。

```ts
// 运行模式选择 + 回退策略
export async function launchAppWithStrategy() {
  const mode = process.env.TEST_LAUNCH_MODE ?? 'launch';
  if (mode === 'cdp') {
    try {
      return await connectOverCDP(process.env.CDP_PORT ?? '9222');
    } catch {
      console.warn('[fallback] cdp 连接失败，回退 launch 模式');
      return await launchByExecutable();
    }
  }
  return await launchByExecutable();
}
```

## 可复用经验

1. 先把“跑得稳”做好，再追求“跑得多”。
2. 覆盖治理要看状态结构，不只看覆盖率总数。
3. 测试治理要有运营视角：台账、优先级、节奏、复盘。
4. 文档和脚本必须成对沉淀，才能持续复用。
