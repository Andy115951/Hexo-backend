## 一句话成就

把“能执行但难解释”的任务能力，升级为“结构可解释、权限可审计、故障可定位”的工程化沙箱体系说明。

---

## 成就卡片

- 成就编号：A03
- 时间范围：2026-01
- 业务域：任务执行架构 / 安全边界治理
- 我的角色：架构拆解 + 权限策略设计 + 故障链路梳理
- 影响对象：研发实现、安全评估、运维排障、新人 onboarding

---

## 背景与问题定义

### 1) 业务场景

任务执行能力覆盖命令运行、文件访问、工具调用和跨进程通信。能力强但风险也高，如果没有边界与治理，问题会从功能缺陷快速演化为安全与稳定性问题。

### 2) 原始痛点

1. 执行链路跨层次，出了问题很难快速定位。
2. 权限边界不清晰，评审时难回答“到底放开了什么”。
3. 故障处理策略零散，超时、取消、输出溢出等场景缺少统一定义。

### 3) 影响评估

- 安全风险：潜在越权与误操作风险难审计。
- 研发效率：跨层排查成本高，问题回归慢。
- 协作成本：产品、研发、安全对能力边界认知不一致。

---

## 目标与验收标准

### 1) 目标

1. 建立端到端执行链路分层模型。
2. 明确默认权限与升级策略，形成可审计边界。
3. 固化关键故障场景处理顺序，提升定位效率。

### 2) 验收标准

- 可用一张分层图解释任务从触发到执行的全链路。
- 可回答“可读/可写/禁止”路径规则以及升级触发条件。
- 对初始化失败、超时、取消、输出截断至少有统一处理策略。
- 新成员可按文档完成一次完整排障流程。

---

## 约束与边界

- 约束 1：默认策略必须最小权限，不以“功能可用”为唯一目标。
- 约束 2：跨平台行为存在差异，需要抽象一致接口。
- 约束 3：执行输出和上下文大小受资源约束，需可控截断。
- 非目标：不在本次引入复杂动态权限系统，优先建立清晰静态边界和升级机制。

---

## 核心原理（为什么这样设计）

1. 默认拒绝 + 最小权限  
执行能力越强，默认边界越要保守，先保证安全下限，再通过显式授权放开能力。

2. 能力授权必须显式可见  
权限升级要可解释、可审计、可追踪，避免“隐式放权”造成边界漂移。

3. 生命周期状态机优先  
初始化、执行、超时、取消、清理必须定义一致语义，避免状态错乱。

4. 可观测性是安全能力的一部分  
没有足够日志和状态输出，安全策略就无法验证，故障也无法复盘。

### 原理流转图（简化）

```text
用户任务请求
  -> 权限预检查（路径/命令/网络）
    -> 通过: 进入执行状态机
       -> running
          -> completed（正常收敛）
          -> timeout（超时终止+状态回写）
          -> cancelled（用户取消+资源回收）
          -> failed（错误分类+诊断信息）
    -> 拒绝: 返回可解释拒绝原因（可申请显式升级）
```

---

## 我的关键动作

## 1) 架构分层与职责切分

我把执行链路拆成四层，避免“全栈黑盒”：

1. 会话与任务编排层：任务创建、状态流转、取消信号。
2. 平台抽象层：屏蔽平台差异，统一接口。
3. 主控进程层：生命周期管理、IPC 协调、执行调度。
4. 沙箱执行层：文件系统、网络、进程、输出约束。

每层都定义“输入、输出、失败语义”，让问题可以按层收敛。

## 2) 权限模型与升级路径梳理

我把权限分成三类：

1. 默认允许：低风险只读能力。
2. 受控允许：白名单路径写入、限定命令前缀。
3. 默认禁止：敏感路径、破坏性操作、高风险网络行为。

同时定义“显式升级”原则：需要升级权限时，必须走可见、可解释、可追踪流程。

## 3) 故障矩阵与恢复策略设计

我把常见异常统一到固定处理框架：

1. 初始化失败：前置检查 + 降级提示。
2. 执行超时：中断进程 + 状态回写 + 诊断信息。
3. 输出过长：截断保关键信息 + 提示获取完整方式。
4. 用户取消：优雅终止 + 资源回收 + 状态一致性保障。

## 4) 文档化与可传递性建设

- 输出“分层图 + 权限矩阵 + 故障顺序”的三件套。
- 把审计问题转成可检查清单，支持评审和复盘。
- 为新成员提供“读图即理解”的入门路径。

---

## 难点、风险与应对

### 1) 难点

- 可用性与安全性天然拉扯：限制太严难用，限制太松不安全。
- 多层协作中容易出现“每层都对但整体失败”的系统性问题。

### 2) 风险与应对

- 风险：隐式放权导致边界漂移。  
  应对：权限升级显式化、流程化。
- 风险：异常处理不统一导致状态不一致。  
  应对：统一状态机语义与恢复顺序。

### 3) 关键突破

- 把“经验规则”升级为“可审计策略”。
- 把“黑盒排障”升级为“分层定位”。

---

## 结果与影响

### 1) 短期结果

1. 执行链路可解释，跨角色沟通成本显著降低。
2. 故障定位有顺序可循，排障效率提升。

### 2) 中期价值

1. 安全评审从口头判断转为策略审计。
2. 后续功能扩展可在既有边界内推进，风险可控。

### 3) 长期复用价值

1. 分层架构与权限模型可迁移到其他执行型能力。
2. “边界先行”的设计方式可成为团队默认工程实践。

```ts
// 权限矩阵（示例）
export const sandboxPolicy = {
  read: ['workspace/**', 'tmp/**'],
  write: ['workspace/output/**', 'tmp/**'],
  deny: ['/etc/**', '/Users/**/.ssh/**', '/System/**'],
  allowCommands: ['node', 'npm', 'pnpm', 'bash', 'sh', 'git'],
  denyCommands: ['rm -rf /', 'mkfs', 'shutdown', 'reboot']
};
```

```ts
export async function executeInSandbox(req: ExecRequest): Promise<ExecResult> {
  assertPathAllowed(req.cwd, sandboxPolicy);
  assertCommandAllowed(req.command, sandboxPolicy);

  const child = spawn(req.command, req.args, { cwd: req.cwd, env: req.env });
  const timer = setTimeout(() => child.kill('SIGTERM'), req.timeoutMs ?? 30000);

  const output = await collectOutput(child, { maxChars: 12000 });
  clearTimeout(timer);

  return {
    code: child.exitCode ?? -1,
    truncated: output.length >= 12000,
    output
  };
}
```

## 可复用经验

1. 高风险能力必须先定义边界，再谈功能拓展。
2. 排障要沿分层路径做证据收敛，避免跳层猜测。
3. 安全策略要可审计、可解释、可执行，三者缺一不可。
4. 复杂系统文档要面向协作，而不是只面向作者本人。
