## 1. 文档目标

本文档基于现有代理配置仓库内容，沉淀一套可复用的方法，用于：

1. 快速搭建可维护的代理分流配置。
2. 对指定上游接口进行“定向弱网注入”。
3. 建立可重复的排障与验证闭环。

---

## 2. 项目结构与职责划分

当前仓库整体是“模板 + 规则 + 模块 + 脚本 + 历史记录”的结构：

- `configs/`：配置模板层
  - `minimal.conf`：最小可用基线
  - `base.conf`：功能完整模板
  - `developer.conf`：开发调试场景模板
- `rules/`：规则数据层
  - `direct.list`：直连目标集合
  - `proxy.list`：代理目标集合
- `modules/`：可插拔能力层
  - 如广告屏蔽模块、定向弱网路由模块
- `scripts/`：自动化控制层
  - 单目标扰动脚本
  - 多目标并行扰动脚本
- 历史文档：记录完整排障过程与关键结论

这个结构的价值是：配置与执行解耦、策略可复用、排障可追踪。

---

## 3. 配置分层范式

建议长期遵循“三层配置法”：

## 第 1 层：基础层（稳定默认）

- 网络基础参数（DNS、接口模式、日志等级）
- 本地网段与局域网直连
- 最终兜底规则

## 第 2 层：场景层（按用途拆分）

- 日常模式
- 开发调试模式
- 弱网验证模式

## 第 3 层：能力层（模块化可开关）

- 广告/跟踪拦截
- 指定域名定向劫持
- 脚本化请求处理

这三层让你在“快速切换场景”和“控制变更风险”之间取得平衡。

---

## 4. 弱网注入架构（核心）

该项目弱网能力的本质是“代理引擎路由 + 故障注入工具”组合：

1. 代理配置把目标域名导向本地监听地址。
2. 本地故障注入服务接收流量并转发到真实上游。
3. 通过 profile 注入延迟、抖动、限速、超时等扰动。

可将 profile 固化为四档：

- `normal`：无扰动（基线）
- `mild`：轻微延迟（观察边缘失败）
- `flaky`：中度抖动 + 少量超时（复现偶发失败）
- `harsh`：高抖动 + 限速 + 高超时（压力场景）

## 5. 标准操作流程（建议）

## 步骤 1：启动

```bash
cd <your-config-repo-path>
./scripts/toxiproxy-chaos-multi.sh up
```

## 步骤 2：切换扰动档位

```bash
./scripts/toxiproxy-chaos-multi.sh profile mild all
# 可改为 normal / flaky / harsh
```

## 步骤 3：查看状态

```bash
./scripts/toxiproxy-chaos-multi.sh status all
```

## 步骤 4：恢复基线

```bash
./scripts/toxiproxy-chaos-multi.sh profile normal all
```

## 6. 排障闭环（从历史问题抽象）

从历史记录看，弱网联调失败通常不是单点故障，而是“多因素叠加”。建议按以下顺序排查：

1. 端口占用检查：低端口是否被其他服务占用。
2. 回环地址检查：是否具备所需本地监听地址。
3. 旧进程/旧容器清理：避免残留冲突。
4. 配置来源检查：是否被托管配置覆盖本地规则。
5. 域名池覆盖检查：目标服务是否存在多域名切换。
6. DNS 路径检查：是否被 fake-IP 或错误解析干扰。
7. 运行模式检查：容器路径与宿主机路径哪个更稳定。
8. 权限检查：低端口绑定是否需要提升权限。

核心原则：先修“链路正确性”，再谈“权限与性能”。

---

## 7. 验证标准（必须有证据）

建议固定三类证据，不凭感觉判断“好了没”：

1. 工具状态证据：`status all` 显示目标代理启用且 profile 生效。
2. 连通性证据：按目标地址做直连/解析验证请求，有可解释响应。
3. 运行日志证据：存在连接建立、转发、收尾的完整日志链路。

---

## 8. 团队落地建议

1. 测试配置与日常配置分离，避免互相污染。
2. 每次变更都保留“基线 profile”回归步骤。
3. 为常见故障维护“最小检查脚本 + 快速修复清单”。
4. 关键参数（延迟、抖动、超时）固定版本化，保证多次实验可对比。

---

## 9. 一句话总结

这个仓库最有价值的不是“某条规则”，而是它形成了一个完整方法：  
**分层配置、定向注入、证据化验证、可追溯排障。**
